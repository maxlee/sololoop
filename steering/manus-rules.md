# Manus 工作流最佳实践

> 来源：Manus 上下文工程理念，适用于复杂多步骤任务

## 2-Action Rule（双动作规则）

每执行 2 次查看/浏览/搜索操作后，立即将关键发现保存到文件。

**目的**：防止视觉/多模态信息在上下文中丢失。

**示例**：
- 浏览了 2 个网页 → 立即记录关键信息到 findings.md
- 查看了 2 个文件 → 立即总结发现到笔记中
- 执行了 2 次搜索 → 立即保存相关结果

## 3-Strike Error Protocol（三振出局协议）

遇到错误时，按以下升级策略处理：

```
尝试 1: 诊断 & 修复
  → 仔细读取错误信息
  → 识别根本原因
  → 应用针对性修复

尝试 2: 替代方案
  → 同样错误？换一个方法
  → 换工具？换库？换实现方式？
  → 绝不重复完全相同的失败操作

尝试 3: 更广泛反思
  → 质疑最初的假设
  → 搜索解决方案
  → 考虑更新整体计划

3 次失败后: 升级给用户
  → 清晰解释尝试过什么
  → 分享具体错误信息
  → 请求指导或澄清
```

**关键原则**：
- 永远不要重复完全相同的失败操作
- 每次尝试必须有实质性的变化
- 记录每次尝试和结果

## 5-Question Reboot Test（五问重启测试）

当上下文可能丢失或需要恢复状态时，确保能回答这 5 个问题：

| 问题 | 答案来源 | 如果答不上来 |
|------|----------|-------------|
| 我在哪里？ | 当前阶段/步骤 | 检查进度记录 |
| 我要去哪里？ | 剩余阶段/目标 | 重读任务计划 |
| 目标是什么？ | 最终目标声明 | 回顾原始需求 |
| 我学到了什么？ | 发现和笔记 | 检查 findings |
| 我做了什么？ | 进度记录 | 检查 git log/历史 |

**使用时机**：
- 长时间暂停后恢复工作
- 上下文窗口接近满载
- 感觉迷失方向时
- 开始新的工作阶段前

## Read vs Write 决策矩阵

| 情况 | 动作 | 原因 |
|------|------|------|
| 刚写完文件 | ❌ 不要立即读 | 内容还在上下文中 |
| 查看了图片/PDF | ✅ 立即写发现 | 多模态内容会丢失 |
| 浏览器返回数据 | ✅ 写入文件 | 截图不持久 |
| 开始新阶段 | ✅ 读计划/发现 | 重新定向 |
| 发生错误 | ✅ 读相关文件 | 需要当前状态来修复 |
| 长时间后恢复 | ✅ 读所有规划文件 | 恢复状态 |

## 反模式（避免）

| ❌ 不要这样做 | ✅ 应该这样做 |
|--------------|--------------|
| 只说一次目标就忘记 | 决策前重读计划 |
| 隐藏错误默默重试 | 记录错误到文件 |
| 把所有内容塞进上下文 | 大内容存文件 |
| 立即开始执行 | 先创建计划文件 |
| 重复失败的操作 | 追踪尝试，变换方法 |
